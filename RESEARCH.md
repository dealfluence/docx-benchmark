Benchmark v2: Authentic Legal Document Scenarios for Autonomous Agent EvaluationThe evaluation of large language models and autonomous agents in legal workflows necessitates a fundamental shift from synthetic, isolated text manipulation to realistic, end-to-end processing of authentic legal instruments. Synthetic documents fail to replicate the labyrinthine Office Open XML (OOXML) structures inherent in heavily negotiated commercial agreements. Current implementations often rely on naive text extraction or simplistic libraries that strip away the underlying structural metadata, rendering the output legally and practically unusable in a real-world corporate environment. Consequently, establishing a robust benchmark suite requires sourcing real-world legal templates that encompass profound semantic density and complex formatting, while remaining unencumbered by restrictive proprietary licenses. This permits their unfettered distribution within an AGPL-3.0 repository.AGPL-3.0 Compatibility and the Mere Aggregation DoctrineIntegrating third-party legal documents into an AGPL-3.0 licensed benchmark repository introduces specific copyright and compatibility considerations that must be navigated with precision. The AGPL-3.0 license applies to the benchmark execution harness, the underlying software, and the evaluation logic. The legal documents themselves, serving as static input data and test fixtures, represent independent works. According to established open-source distribution principles, placing distinct, independently licensed works alongside software on a distribution medium constitutes "mere aggregation".The mere aggregation doctrine ensures that the AGPL-3.0 license does not infect or govern the independent data files, nor does it require the data files to be relicensed under the AGPL. However, for the repository to remain entirely open and friction-free for enterprise adoption, the data files must possess licenses that permit redistribution and use in both commercial and non-commercial contexts, aligning with the overarching Open Source Definition.Licenses such as the Creative Commons Attribution 4.0 International (CC BY 4.0) and the UK Open Government Licence v3.0 (OGL v3.0) permit unrestricted commercial redistribution, modification, and use, provided attribution requirements are met. These licenses are inherently compatible with aggregation in an open-source repository. Similarly, the Creative Commons Zero (CC0) public domain dedication removes all copyright barriers, offering maximum interoperability.Conversely, documents carrying "No Derivatives" (ND) clauses, such as the CC BY-ND 4.0 license, strictly prohibit the public distribution of modified versions. In the context of an autonomous agent benchmark, the repository distributes the unmodified original file as a test fixture. The dynamic modification of the file is performed locally or in-memory by the agent during the test execution, which constitutes private use rather than the public distribution of a derivative work. Therefore, CC BY-ND 4.0 documents may be safely included as input fixtures, provided the benchmark harness does not distribute the agent's modified output publicly.Documents bearing strict "Non-Commercial" restrictions conflict fundamentally with the freedom to use open-source software for any purpose and must be excluded from the repository. For example, the World Bank provides highly structured Standard Procurement Documents (SPDs), but their licensing explicitly dictates that any commercial use, reselling, or redistribution is strictly prohibited. Agents that process documents by converting them to plain text or markdown inevitably destroy these underlying structures. Systems like safe-docx and OpenAgreements have been engineered specifically to manipulate these files surgically via Model Context Protocol (MCP) tools, preserving formatting, tracked changes, and comments. The benchmark must stress-test these capabilities, penalizing destructive operations that sever XML linkages.Footnotes and endnotes represent a significant structural hurdle. In the OOXML standard, footnote content does not reside within the main document body (document.xml); rather, it lives in a separate XML part (footnotes.xml). The main document utilizes <w:footnoteReference> tags to link to the corresponding text. When an autonomous agent is instructed to perform a structural edit, a naive text replacement algorithm will often fail to locate the text inside the separate footnote part. Furthermore, deleting a reference in the main body without programmatically cleaning the associated XML part leaves orphaned data and causes document corruption.Cross-references and bookmarks pose an equally pervasive challenge. Highly negotiated contracts rely heavily on defined terms and internal cross-references constructed using field codes (<w:instrText> REF </w:instrText>) that point to specific bookmark tags (<w:bookmarkStart> and <w:bookmarkEnd>) wrapping the referenced text. If an agent replacing a defined term overwrites the invisible bookmark tags, all subsequent references to that term will display error codes upon rendering, demonstrating a failure in the agent's contextual awareness of the Abstract Syntax Tree (AST).Automatic heading numbering in legal documents relies on intricate XML definitions where individual paragraphs contain a <w:numId> that points to complex <w:abstractNum> list definitions stored in numbering.xml. Legal numbering schemes (e.g., 1.1, 1.1.1, 1.2) are fragile. If a "Playbook-based Commenting" scenario requires an agent to delete an entire representation section, the agent must understand how to maintain the continuity of the subsequent list elements. A destructive edit will cause the underlying XML references to crash or restart incorrectly, destroying the document's visual and structural hierarchy.These numbering schemes are inextricably linked to field-based Tables of Contents (TOC). A TOC in Word is a dynamic field code that references specific heading styles. Any structural edits, such as inserting or deleting sections, invalidate the TOC. Agents must demonstrate the capability to either execute an update field command via an API or programmatically recalculate pagination and heading strings to repair the TOC post-edit.Finally, nested tables and Content Controls (Structured Document Tags or <w:sdt>) fundamentally alter editing semantics. Nested tables, common in compliance schedules, create hierarchies of <w:tbl>, <w:tr>, and <w:tc> elements, often with merged cells that make positional editing highly ambiguous. Content Controls wrap specific runs of text to provide form-field behavior, requiring agents to understand the semantic difference between editing raw text (<w:t>) and interacting with the <w:sdtContent> payload to preserve template data bindings.Analysis of Primary Open-Source Legal EcosystemsThe open-source legal landscape is dominated by specialized standard agreements designed to reduce transaction friction. These document suites are optimized for different segments of commercial law, providing a rich diversity of structural and semantic test vectors.The Common Paper StandardsCommon Paper has engineered a suite of commercial agreements specifically optimized for modern, programmatic contracting workflows. Created by a committee of over forty attorneys representing technology vendors, procurement teams, and major law firms, the portfolio includes Mutual Non-Disclosure Agreements, Cloud Service Agreements (CSAs), Service Level Agreements (SLAs), Data Processing Agreements (DPAs), and Design Partner Agreements. Released entirely under the CC BY 4.0 license, these documents are explicitly designed to be free to use, modify, and redistribute, making them ideal candidates for the benchmark repository.The architectural brilliance of Common Paper templates lies in their bifurcation into a "Cover Page" and "Standard Terms". The Cover Page contains all heavily negotiated commercial variables, formatted as bracketed placeholders or structured fill-in fields, while the Standard Terms remain static and are incorporated by reference or appended as a separate section. This distinct structure is highly suited for evaluating "Form Fill" operations. Agents must demonstrate the capability to parse JSON-formatted deal data, identify the corresponding bracketed placeholders in the Cover Page .docx file, and perform surgical string replacements. The benchmark will stress-test the agent's ability to disambiguate identical underscore patterns (e.g., [____________]) by analyzing the surrounding contextual text to ensure the correct variable is inserted without invalidating the document's styles. Furthermore, Common Paper’s modular approach, where a CSA is frequently executed alongside an SLA and a DPA, provides a natural, pre-aligned fixture set for testing cross-document awareness and variable propagation across a multi-file assembly.The Bonterms Standard AgreementsParallel to Common Paper, Bonterms offers a library of standardized commercial contracts, drafted by an extensive committee of over 120 legal professionals, and distributed under the CC BY 4.0 license. The Bonterms architecture also utilizes the Cover Page methodology to isolate negotiated variables but integrates a significantly higher degree of internal complexity within the .docx files themselves.Bonterms documents are heavily reliant on internal cross-references, deep schedule attachments, and dense footnotes that govern the interpretation of specific indemnification and liability clauses. When an autonomous agent is tasked with a "Contract Clone & Party Swap" workflow, it must locate and replace party names and defined terms throughout the Bonterms Cloud Terms. Bonterms fixtures effectively measure an agent's capability to manipulate the AST securely, preserving the integrity of field codes and <w:footnoteReference> tags while altering the surface text. The complex interplay between the Base Terms and the attached Service Level Agreements provides an optimal environment for evaluating sophisticated redlining and markup insertion via MCP tool calls.National Venture Capital Association (NVCA) Model DocumentsThe National Venture Capital Association (NVCA) model documents represent the absolute zenith of complexity in private market transactional law. This suite, comprising the Term Sheet, Stock Purchase Agreement (SPA), Investors' Rights Agreement (IRA), Right of First Refusal and Co-Sale Agreement (ROFR), and Voting Agreement, forms the legal foundation for the vast majority of venture capital financings in the United States. While copyrighted by PitchBook Data, Inc., the documents are universally recognized as "free to use" and have been distributed openly for decades to establish industry norms, permitting their inclusion in the repository as unmodified test fixtures.The NVCA suite is indispensable for the benchmark due to its extraordinary structural and semantic density. The documents contain profound legal mechanics, such as participating versus non-participating liquidation preferences, broad-based weighted average anti-dilution protections, and highly specific representations and warranties. Recent updates to the NVCA forms reflect emerging market realities, including sophisticated representations regarding the use of Generative AI, open-source software compliance, and stringent national security and export control oversight.From a structural OOXML perspective, NVCA documents feature intricate, multi-level automatic heading numbering schemes and deeply nested schedules. In a "Playbook-based Commenting" scenario, an agent might be instructed to evaluate the "Company-Controlled Intellectual Property" representations based on a buyer-friendly playbook. Modifying or deleting a numbered section in the SPA necessitates that the agent understands how to maintain the continuity of the subsequent list elements without corrupting the underlying XML <w:num> hierarchy. The interconnected nature of the SPA, IRA, and ROFR also serves as the ultimate test for multi-file deal assembly, requiring the agent to synthesize capitalization table data and distribute it flawlessly across signature blocks and schedules spanning five distinct, highly formatted files.Y Combinator Simple Agreement for Future Equity (SAFE)The Y Combinator SAFE, introduced in 2013 and comprehensively updated to a "post-money" format in 2018, revolutionized early-stage startup financing by replacing debt-based convertible notes with a standardized, high-resolution equity derivative. The SAFE documents are distributed freely by Y Combinator, and while they are subject to CC BY-ND 4.0 licensing, their inclusion as unmodified read-only input fixtures in an open-source repository complies seamlessly with standard fair use and mere aggregation doctrines.The YC SAFE provides a distinct testing vector for the benchmark suite. While the .docx structure is relatively straightforward compared to the NVCA suite, the semantic meaning of the text is mathematically and legally rigid. The documents are offered in specific variants: Valuation Cap with No Discount, Discount with No Valuation Cap, and an Uncapped Most Favored Nation (MFN) version. The "Form Fill" scenario utilizing a SAFE requires the agent to accurately format financial figures, calculation mechanics, and percentages within the cap and discount fields without disturbing the surrounding boilerplate. Furthermore, because the SAFE is intended to be used without structural modification, it serves as an excellent fixture for "Policy Checklist Review." An agent can be fed an executed SAFE and tasked with determining whether the instrument operates on a pre-money or post-money basis, whether it includes a pro-rata side letter, and the precise valuation cap applied, testing the agent's ability to extract nuanced financial reasoning from standardized legal text.Series Seed DocumentsAs an alternative to the comprehensive but heavy NVCA framework, the Series Seed documents offer a standardized set of legal agreements tailored specifically for smaller, earlier-stage equity rounds. Critically, the Series Seed documents, originally crafted by Fenwick & West, are explicitly dedicated to the public domain under the CC0 license, entirely eliminating any friction regarding derivative works, redistribution, or commercial use.Hosted natively on GitHub, the Series Seed suite includes a Term Sheet, a Preferred Stock Investment Agreement, and an Amended and Restated Certificate of Incorporation. These documents encapsulate many of the same legal concepts as the NVCA forms, such as board composition rights and conversion mechanics, but in a significantly condensed format. They are highly appropriate for the "Contract Clone & Party Swap" scenario. Given their CC0 status, the benchmark harness can freely generate and distribute extensively modified, synthetic versions of the Series Seed documents to test whether agents can identify anomalous edits and revert them back to the standard baseline.Public Sector and Regulatory FrameworksTo ensure the benchmark comprehensively evaluates agentic capabilities across diverse legal domains, public sector procurement and international regulatory compliance documents must be integrated. These fixtures introduce massive scale and specialized formatting requirements not typically found in startup-focused venture documents.UK Government Model Services ContractThe Model Services Contract (MSC), published by the UK Cabinet Office and the Government Legal Department, governs complex, high-risk public sector IT and business process outsourcing procurements valued at over £20 million. The MSC is released under the Open Government Licence v3.0 (OGL v3.0), which is fundamentally compatible with the AGPL-3.0 repository environment, permitting unfettered commercial reuse and modification with simple attribution.The structural magnitude of the MSC is unparalleled in this dataset. It is divided into Core Terms and extensive Schedules covering performance indicators, security management, insurance requirements, and data protection. The .docx files feature highly complex field-based Tables of Contents (TOC) that rely on heading style mappings. Furthermore, the Schedules are dominated by deeply nested tables utilized for defining Service Credits, "Target Cost" pricing mechanisms, and Key Performance Indicators (KPIs). For the benchmark, the MSC is the definitive fixture for testing OOXML Table of Contents invalidation. If an agent inserts or deletes sections during a "Playbook-based Commenting" or structural editing scenario, it must programmatically update the TOC field codes to reflect the new pagination and heading structures. The sheer length of the document—often exceeding hundreds of pages—also stress-tests the context window retrieval accuracy and token-efficiency of the underlying language models driving the agents.European Union Standard Contractual Clauses (SCCs)The transfer of personal data outside the European Economic Area under the General Data Protection Regulation (GDPR) is heavily reliant on the Standard Contractual Clauses (SCCs). The European Commission publishes the official .docx versions of the SCCs, and their reuse is authorized under a policy equivalent to the CC BY 4.0 license, facilitating seamless integration into the benchmark.The SCCs are constructed using a highly modular format containing four distinct transfer modules (e.g., Controller-to-Processor, Processor-to-Processor) and require the completion of specific Annexes detailing the parties, the description of the transfer, and the technical and organizational security measures. These Annexes are presented as rigid tables within the .docx file. The SCCs serve as the optimal fixture for the "Policy Checklist Review" scenario. An agent can be provided with a completed SCC document and a compliance checklist requiring verification of the specific module selected, the inclusion of the competent supervisory authority, and the adequacy of the security measures detailed in Annex II. It is recommended that a dedicated /fixtures directory be established, subdivided meticulously by source organization (e.g., /fixtures/nvca, /fixtures/common-paper, /fixtures/uk-gov-msc). Every subdirectory must contain a LICENSE or README.md file explicitly detailing the provenance of the documents, the governing license framework (CC BY 4.0, CC0, or OGL v3.0), and the requisite attribution notices acknowledging the original authors, fulfilling all copyleft and open-source compliance standards.For documents utilized under the doctrine of mere aggregation, such as the Y Combinator SAFE templates, the repository must ensure that the master fixture files remain strictly unmodified in the codebase. Any automated modifications generated by the autonomous agents must occur entirely within the transient, local execution environment of the benchmark harness to avoid the public distribution of unauthorized derivative works. By securing this diverse array of highly complex, openly licensed legal documents and deploying them across the rigorous scenario matrix defined above, Benchmark v2 will possess the necessary fidelity to accurately evaluate the formatting preservation, structural comprehension, and semantic reasoning capabilities of autonomous legal agents in authentic, high-stakes commercial environments.
